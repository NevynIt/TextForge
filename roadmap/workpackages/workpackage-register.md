# Workpackage Register — Roadmap V19a

**Status:** Canonical mutable WP5+ planning register  
**Initialized from:** V18 dependency reassessment, cleaned and extended in V19, sharpened in V19a  
**Scope:** Active WP5+ roadmap. Frozen baseline work remains listed for dependency tracking, but new scope must be added as follow-on workpackages.

V19a keeps archive/source-roadmap material out of the active package and records the selected path: ITM visual recovery first, then the minimal BPMN visual consumption chain. Use this register, not old phase files, as the scheduling authority.

## Status vocabulary

| Status | Meaning |
|---|---|
| Ready / next | Suitable next implementation target. |
| Not started | Defined but not yet implemented. |
| In progress | Actively being implemented. |
| Implemented | Code/docs changed and basic verification passed. |
| Validated | Implementation evidence satisfies acceptance criteria. |
| Deferred / standalone | Not blocking generic roadmap progress. |
| Candidate | Possible future workpackage; not yet committed. |
| Candidate / standalone | Candidate that can proceed independently once its direct dependency is met. |
| Blocked | Cannot progress until dependency or decision clears. |

## Current reading

- The active roadmap is V19. Do not use removed archive/source-roadmap files as implementation targets.
- Frozen validated baseline: `WP-05A`, `WP-05B`, `WP-05C`, `WP-05D`, `WP-RES-01`, `WP-REPO-01`, `WP-ITM-01`, `WP-ITM-02`, `WP-ITM-VISUALS`, `WP-LUA`, and `WP-LUA-POWER-SESSION`.
- `WP-ITM-VISUALS` is static projection/publication baseline, not full runtime renderer parity.
- Recommended V19a main chain: `WP-VITM-01 -> WP-ITM-VTARGET-01 -> WP-ITM-VRESOLVE-01 -> WP-RENDER-CYTOSCAPE -> WP-RENDER-JSMIND -> WP-RENDER-SIGMA -> WP-BPMN-SEM -> WP-BPMN-VISUAL-A -> WP-BPMN-VISUAL-B`.
- `WP-VITM-TRANSLATORS` depends only on `WP-VITM-01` and can be scheduled independently of visual editing.
- `WP-GRAPH-EDIT-VITM` and `WP-BPMN-VISUAL-C` are later visual editing/write-back infrastructure and should not block read-only visual consumption parity.
- `WP-MD-REPORT`, `WP-ARCHIMATE-SEM`, `WP-TABLES`, `WP-RES-02`, `WP-ID-01`, and `WP-SET-01` remain separately startable under the current dependency posture, but they are not on the selected minimal BPMN visual chain.
- `WP-BPMN-VISUAL-A` should use BPMN.io / `bpmn-js`; `WP-BPMN-VISUAL-B` integrates that surface with ITM visual-target resolution; `WP-BPMN-VISUAL-C` is deferred editing/write-back.
- Entra SSO remains a standalone production adapter and does not block generic backend development.


## V19a definition and grilling gates

- `WP-VITM-01` definition is now drafted in `specs/architecture/visual-itm-v1-profile.md`; implementation can proceed against that profile.
- `WP-ITM-VRESOLVE-01` grilling is now captured in `grilling/v19a-visual-recovery-to-bpmn-chain-findings.md`; implementation can proceed against that resolver contract.
- `WP-BPMN-SEM` needs grilling/further definition before implementation to lock a minimal BPMN MVP subset and avoid drifting into full BPMN coverage.
- `WP-VITM-VDELTA-01` and `WP-VITM-LIVE-SYNC-01` now exist as explicit later follow-on workpackages. They are not blockers for `WP-VITM-01`, `WP-ITM-VTARGET-01`, or `WP-ITM-VRESOLVE-01`.
- `WP-BPMN-VISUAL-A/B/C` are the replacement split for the former monolithic `WP-BPMN-VISUAL`; only A and B belong to the minimal BPMN visual consumption chain.
